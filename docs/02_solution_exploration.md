# 해결방안 탐색서 (Solution Exploration)

> **문서 용도:** 이 문서는 문제정의서에서 정의한 사용자 행동 루프(Store Loop, Confirm Loop, Total Loop)의 병목을 해결하기 위한 아이디어를 정리한 탐색 문서다.  
> MVP 단계에서는 모든 아이디어를 구현하는 것이 아니라, 효과가 검증 가능한 구조를 우선적으로 탐색한다.  
> 각 해결안의 채택/배제 판단은 결정로그 문서(03_decision_log.md)에서 결정한다.

---

## 1. 문서 목적

- 문제정의서에서 정의한 병목 구간(Store Loop, Confirm Loop, Total Loop)을 해결하기 위한 방안을 구조적으로 탐색한다.
- 각 해결안이 **어떤 사용자 문제를 해결하는지**, **어떤 지표에 영향을 주는지**를 명확히 한다.
- 결정로그에서 채택/배제 판단을 하기 위한 기준 자료를 제공한다.

---

## 2. Store Loop 개선 아이디어 (SED-1-b)

### 2.1 문제 요약

- 사용자는 여행 준비 정보를 탐색하더라도, 구조화되지 않은 정보로 인해 필요한 내용을 한 번에 확보하지 못하고 준비 과정이 지연된다.
- `Detail Check` 이후에도 바로 `Save`로 이어지지 않아 반복 탐색이 발생한다.

**관련 루프:**
```
Search → Detail Check → Save
```

**관련 지표:**

| 지표 | 공식 |
|------|------|
| `SP1` | `DetailCheck Sessions / Search Sessions` |
| `SP2` | `Save Sessions / DetailCheck Sessions` |
| `Store_LCP` | `SP1 × SP2` |
| `Store_R` | `ReSearch Sessions / Search Sessions` |
| `Delay_t` | 검색→저장 평균 소요 시간 (정규화) |
| `PB_t` | 준비 부담 지수 |

---

### 2.2 해결안 1: 맞춤형 준비 항목 추천 → SP1 개선

**가설**
> [여행을 준비중인 사용자]가 [자신의 조건에 딱 맞는 정보를 찾기 위해 반복 탐색하는 상황]에서  
> [조건 필터를 활용해 맞춤형 탐색 결과를 제공한다면]  
> `Store_R`이 감소하여 `PB_t`가 줄고, `SP1`이 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `SP1` | ↑ 증가 |
| `Store_R` | ↓ 감소 |
| `PB_t` | ↓ 감소 |

**개발 관점 메모**
- 여행 조건 입력 UI (여행지, 일정, 동행인 등) 필요
- 조건 → 항목 추천 반환 API 연동 필요
- 필터 조건이 너무 적으면 추천 정확도 낮아질 수 있음

---

### 2.3 해결안 2: 즉시 저장 가능한 체크리스트 구조 → SP2 개선

**가설**
> [정보를 탐색 중인 사용자]가 [찾은 정보를 다시 체크리스트로 정리하여 저장해야 하는 상황]에서  
> [최소한의 클릭으로 즉시 체크리스트로 저장 가능한 구조를 제공한다면]  
> `Delay_t`가 감소하고, `PB_t`가 줄어 `SP2`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `SP2` | ↑ 증가 |
| `Delay_t` | ↓ 감소 |
| `PB_t` | ↓ 감소 |

**개발 관점 메모**
- Save 버튼 클릭 → 즉시 DB 저장 구조 필요
- 별도 변환 또는 정리 과정 없이 바로 리스트에 추가되어야 함
- 저장 행동이 늘어도 실제 준비 행동까지 연결 보장은 불가 (측정 한계)

---

### 2.4 해결안 3: 탐색 결과 구조화 제공 → Detail Check 개선

**가설**
> [여행을 준비중인 사용자]가 [자신의 조건에 맞는 정보를 탐색하는 상황]에서  
> [사용자 조건에 맞는 구조화된 항목을 제공한다면]  
> `Store_R`과 `Delay_t`가 감소하고, `PB_t`가 줄어 `SP2`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `SP2` | ↑ 증가 |
| `Store_R` | ↓ 감소 |
| `Delay_t` | ↓ 감소 |
| `PB_t` | ↓ 감소 |

**개발 관점 메모**
- 결과 데이터를 체크리스트 형태로 가공하여 제공해야 함
- FE에서 바로 렌더링 가능한 구조 필요
- 기반 데이터 부족 시 추천 정확도 저하 가능

---

### 2.5 Store Loop 장단점 정리

**장점**

| 효과 | 관련 해결안 |
|------|-------------|
| 검색 착수율 개선 가능 | 1 |
| 저장 전이 향상 가능 | 2, 3 |
| `Delay_t` 감소 가능 | 2, 3 |
| 반복 탐색 감소 가능 | 1, 3 |
| Store Loop 안정성 향상 가능 | 1, 2, 3 |
| 루프 완성 확률 향상 | 1, 2 |

**단점**

| 리스크 | 관련 해결안 |
|--------|-------------|
| 필터 조건이 적으면 맞춤 결과 제공 어려움 | 1 |
| 필터 조건 복잡 시 첫 저장 전 이탈 증가 가능 | 1 |
| 기반 데이터 부족 시 추천 정확도 낮음 | 1, 3 |
| 저장 행동 증가 = 실제 준비 행동 보장 아님 | 2 |

---

### 2.6 Store Loop 관찰 포인트

**Leading (선행 지표)**
- `Detail Check` 후 첫 `Save` 발생률
- `Search` 후 첫 `Detail Check`까지 걸리는 시간
- `Detail Check` 후 첫 `Save`까지 걸리는 시간
- `Search` 이후 `Detail Check` 발생 여부
- `Detail Check` 이후 `Save` 발생 여부

**Structural (구조 지표)**
- `SP1`, `SP2`, `Store_LCP`, `Store_R`
- `Search` → `Detail Check` 평균 소요 시간
- `Detail Check` → `Save` 평균 소요 시간
- 검색 착수율 (`SS`)
- 저장 전환율 (`SCR`)

**Lagging (후행 지표)**
- `PB_t` 변화
- `Store_LSI` 변화

## 3. Confirm Loop 개선 아이디어 (SED-2-b)

### 3.1 문제 요약

- 사용자는 저장한 정보를 다시 열어봐도 실제 준비 행동까지 바로 이어가지 못한다.
- 점검 과정에서 빠진 항목을 발견하면 다시 Store Loop로 돌아간다 (Backflow 발생).

**관련 루프:**
```
Saved List Open → Edit (Edit_text / Edit_add / Edit_del / Edit_reorder) → Prepare Action
```

**관련 지표:**

| 지표 | 공식 |
|------|------|
| `CP1` | `Edit Sessions / SavedListOpen Sessions` |
| `CP2` | `PrepareAction Sessions / Edit Sessions` |
| `Confirm_LCP` | `CP1 × CP2` |
| `Backflow` | `ReStoreTrigger Sessions / SavedListOpen Sessions` |

---

### 3.2 해결안 1: 실행 순서 중심 체크리스트 구조 → CP1 개선

**가설**
> [저장한 준비 정보를 다시 확인하는 사용자]가 [저장 리스트를 열어봐도 무엇부터 점검해야 할지 바로 판단하지 못하는 상황]에서  
> [실행 순서 중심 체크리스트 구조를 제공한다면]  
> 단순 열람 행동을 줄이고 Edit 행동이 증가하여 `CP1`이 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `CP1` | ↑ 증가 |
| `CP2` | ↑ 증가 |

---

### 3.3 해결안 2: 카테고리 기반 확인 구조 → CP1 개선

**가설**
> [여행 준비 리스트를 확인하는 사용자]가 [저장된 항목을 스스로 다시 분류하며 점검해야 하는 상황]에서  
> [준비물·예약·출국 전 확인사항 등 카테고리 기반 확인 구조를 제공한다면]  
> 비구조적 확인 행동을 줄이고 점검/수정 행동이 증가하여 `CP1`이 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `CP1` | ↑ 증가 |
| Edit 행동 | ↑ 증가 |

**개발 관점 메모**
- `category` 필드 DB 스키마 필요
- 카테고리 탭/섹션 UI 구성 필요

---

### 3.4 해결안 3: 누락 항목 추천 → Backflow 감소

**가설**
> [저장 리스트를 다시 확인하는 사용자]가 [빠진 준비 항목을 뒤늦게 발견해 다시 검색으로 돌아가는 상황]에서  
> [누락 가능 항목을 추천해준다면]  
> 확인 중 재탐색 행동이 줄어 `Backflow`가 감소하고 `Confirm_LSI`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `Backflow` | ↓ 감소 |
| `Confirm_LSI` | ↑ 증가 |

**개발 관점 메모**
- MVP 초기에는 rule 기반 추천으로 시작 가능 (ex. 여행지별 필수 준비물)
- 추천이 부정확하면 오히려 불필요한 수정·재탐색 발생 가능
- 추천 항목이 너무 많으면 사용자 피로도 증가

---

### 3.5 해결안 4: 휴지통 기능 → Backflow 감소

**가설**
> [저장한 리스트를 확인하는 사용자]가 [삭제한 체크리스트 항목을 다시 재탐색·재저장해야 하는 상황]에서  
> [삭제 항목을 다시 확인·복구할 수 있는 휴지통 기능을 제공한다면]  
> `Backflow`가 감소하고 `Confirm_LSI`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `Backflow` | ↓ 감소 |
| `Confirm_LSI` | ↑ 증가 |

**개발 관점 메모**
- soft delete 구조 필요 (DB에서 즉시 삭제하지 않고 `deleted_at` 필드 관리)
- 휴지통 UI 별도 탭 or 모달로 제공

---

### 3.6 해결안 5: 준비 완료 체크 행동 강화 → CP2 개선

**가설**
> [저장 리스트를 점검한 사용자]가 [준비 진행 과정을 명확히 확인하기 어려운 상황]에서  
> [준비 완료된 항목을 쉽게 확인할 수 있는 구조를 제공한다면]  
> 단순 열람 행동이 줄고 `Prepare Action`으로 이어져 `CP2`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `CP2` | ↑ 증가 |
| Prepare Action | ↑ 증가 |

---

### 3.7 Confirm Loop 장단점 정리

**장점**

| 효과 | 관련 해결안 |
|------|-------------|
| Edit 행동 유도 가능 | 1, 2 |
| Prepare Action 전환 가능성 증가 | 5 |
| 재탐색 비율 감소 가능 | 3, 4 |
| Confirm Loop 안정성 향상 | 3, 4 |
| 루프 완성 확률 향상 | 5 |
| Edit 데이터 → 추후 추천 데이터셋으로 활용 가능 | 1, 2 |

**단점**

| 리스크 | 관련 해결안 |
|--------|-------------|
| 점검 요소가 많아지면 확인 루프 자체가 복잡해질 수 있음 | 1, 2, 3 |
| 추천 부정확 시 불필요한 수정·재탐색 발생 | 3 |
| 추천 항목 과다 시 선택 피로도 증가 | 3 |
| 체크 구조 강화 = 실제 준비 완료 보장 아님 | 5 |
| 확인 기능이 많아질수록 UI 부담 증가 | 전반 |

---

### 3.8 Confirm Loop 관찰 포인트

**Leading (선행 지표)**
- `Saved List Open` 후 Edit 전환률 / 미전환률
- 실행 순서 중심 체크리스트 사용률
- 카테고리별 점검 진입률
- 누락 항목 추천 클릭률
- 체크/수정 기능 사용률
- 휴지통 기능 사용률
- `Saved List Open` 이후 Edit 발생 여부
- Edit 이후 Prepare Action 발생 여부

**Structural (구조 지표)**
- `CP1`, `CP2`, `Confirm_LCP`, `Backflow`
- 준비 완료(전체 체크)까지 걸리는 시간
- Edit 내 세부 행동별 평균 사용률
  - `Edit_text` / `Edit_add` / `Edit_del` / `Edit_reorder`

**Lagging (후행 지표)**
- `Confirm_LSI` 변화
- `Saved List Open` 대비 `Prepare Action` 도달률
- 저장 이후 `Saved List Open` 전환율
- 코호트별 Edit 사용 비율
- 코호트별 `Saved List Open` 비율

## 4. Total Loop 개선 아이디어 (SED-3-b)

### 4.1 문제 요약

1. 여행이 확정된 후 Store Loop까지 이어지는 진입 구조가 매끄럽지 못해 탐색 시간 비용이 증가한다.
2. 확인 루프에서 실행 완료 이후 서비스를 다시 사용하려는 재사용 동기가 부족하다.
3. 그 결과 전체 준비 구조가 하나의 안정적인 루프로 닫히지 못하고 이탈 또는 역류가 발생한다.

**관련 루프:**
```
Travel Fixed → [Store Loop] → [Confirm Loop] → Reuse
```

**관련 지표:**

| 지표 | 공식 |
|------|------|
| `Total_LCP` | `Store_LCP × Confirm_LCP` |
| `Total_LSI` | `Total_LCP × (1 - λ1·Store_R - λ2·Backflow)` |

---

### 4.2 해결안 1: 준비 시작 진입 구조 도입 → Travel Fixed → Store Loop 전환 개선

**가설**
> [여행을 준비중인 사용자]가 [여행이 확정된 이후에도 어디서부터 시작해야 할지 몰라 다시 탐색부터 시작하는 상황]에서  
> [바로 체크리스트를 생성할 수 있는 준비 시작 구조를 제공한다면]  
> 무계획 탐색 행동이 줄고 초기 진입 마찰이 감소하여 `Store_LCP`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `Store_LCP` | ↑ 증가 |
| Travel Fixed → Store Loop 전이율 | ↑ 증가 |

**개발 관점 메모**
- 초기 진입 화면 / 온보딩 UI 필요
- 체크리스트 생성 트리거 설계 필요 (여행 추가 버튼 등)

---

### 4.3 해결안 2: 저장-확인 루프 연결 구조 → Store → Confirm 연결 개선

**가설**
> [여행 준비 서비스를 사용하는 사용자]가 [저장한 정보와 이후 점검 행동이 분리되어 확인 루프 이탈·재진입을 반복하는 상황]에서  
> [저장된 항목이 바로 확인 루프로 이어지는 구조를 제공한다면]  
> 저장 후 이탈 행동이 줄고 `Saved List Open` 및 Edit 행동이 증가하여 `Total_LCP`가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `Total_LCP` | ↑ 증가 |
| Store → Confirm 전이율 | ↑ 증가 |

---

### 4.4 해결안 3: 진행도 시각화 → 반복 확인 및 Backflow 감소

**가설**
> [여행을 준비 중인 사용자]가 [현재 어디까지 준비했는지 명확히 알지 못해 같은 내용을 반복 확인하는 상황]에서  
> [준비 단계별 진행도를 시각화한다면]  
> 중복 확인 행동이 줄고 필요한 항목 중심의 점검 행동이 증가하여 `Backflow`와 반복 확인이 감소할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| `Backflow` | ↓ 감소 |
| 반복 확인 행동 | ↓ 감소 |
| `Total_LCP` | ↑ 증가 |

---

### 4.5 해결안 4: Reuse 동기 설계 → 장기 재활성화

**가설**
> [여행 준비 서비스를 한 번 사용한 사용자]가 [한 번의 여행 준비 이후 서비스를 종료하는 상황]에서  
> [다음 여행에도 다시 사용할 이유가 보이는 구조를 제공한다면]  
> 일회성 사용 행동이 줄고 새로운 Store Loop를 다시 시작하는 행동이 늘어나 Reuse 전이가 증가할 것이다.

**기대 효과**

| 지표 | 방향 |
|------|------|
| Reuse 전이율 | ↑ 증가 |
| 새로운 Store Loop 재발생 | ↑ 증가 |

> **주의:** Reuse를 초기에 과도하게 고려하면, 1차 MVP의 핵심 목표인 준비완료까지의 사용자 행동 루프 관찰이 흐려질 수 있다.

---

### 4.6 Total Loop 장단점 정리

**장점**

| 효과 | 관련 해결안 |
|------|-------------|
| 루프 간 단절 감소 (루프 완성율 향상) | 2, 3 |
| 저장만 하고 끝나는 비율 감소 | 2, 3 |
| 전체 흐름을 하나의 구조 안에서 관찰 가능 | 2, 3 |
| 반복 확인·역류 감소 | 2, 3 |
| Reuse를 통한 장기 유지율 연결 가능 | 4 |
| 전체 루프 안정성 향상 | 1, 2 |
| 초기 진입 마찰 감소 | 1 |

**단점**

| 리스크 | 관련 해결안 |
|--------|-------------|
| MVP에서 너무 많은 단계를 통합하면 구조 복잡화 | 2, 3, 4 |
| 통합 제공 시 지표 수집이 어려울 수 있음 | 2 |
| 진행도 시각화 = 실제 준비 행동 보장 아님 | 3 |
| Reuse 초기 과도 반영 시 핵심 관찰 목표 흐려짐 | 4 |
| 이벤트 정의 변경 시 지표 비교 가능성 낮아짐 | 전반 |

### 4.7 Total Loop 관찰 포인트

**Leading (선행 지표)**
- Travel Fixed 이후 첫 Search 발생률
- Travel Fixed 이후 첫 Save 발생률
- Save 이후 첫 `Saved List Open` 발생률
- Prepare Action 이후 Reuse 진입률
- Travel Fixed 이후 Store Loop 발생 여부
- Backflow 발생 여부
- Confirm Loop 이후 Reuse 발생 여부

**Structural (구조 지표)**
- Travel Fixed → Store Loop 전이율
- Store Loop → Confirm Loop 전이율
- `Backflow`
- `Total_LCP`

**Lagging (후행 지표)**
- `Total_LSI`
- Prepare Action → Reuse 전이율
- 새로운 Store Loop 발생률
- 코호트별 Reuse 비율

## 5. MVP 개발 관점 정리

### 기능 추가 판단 기준

```
이 기능이 어느 루프의 어떤 지표를 개선하는가?
→ Store_LCP를 높이는가?
→ Backflow를 줄이는가?
→ Total_LCP를 높이는가?
```

### 루프별 우선순위 정리

| 루프 | 핵심 병목 | 우선 해결안 |
|------|-----------|-------------|
| **Store Loop** | Search → Save 전환 마찰 | 즉시 저장 구조, 구조화된 탐색 결과 제공 |
| **Confirm Loop** | Edit → Prepare Action 단절, Backflow | 카테고리 구조, 완료 체크 강화, 휴지통 |
| **Total Loop** | 루프 전체 연결 단절 | 준비 시작 진입 구조, 저장-확인 연결 |

---

## 6. 이 문서의 핵심

> - **Store Loop** → "검색 → 저장" 전환을 끊기지 않게 만드는 것  
> - **Confirm Loop** → "저장 → 실행" 전환을 만들고 Backflow를 줄이는 것  
> - **Total Loop** → 전체 흐름이 하나로 연결되도록 만드는 것
>
> MVP에서는 모든 것을 해결하려 하지 않는다.  
> **가장 큰 병목을 줄일 수 있는 최소 기능을 선택하는 것이 핵심이다.**
