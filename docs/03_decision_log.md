# 결정로그 (Decision Log)

> **문서 용도:** 이 문서는 해결방안 탐색서(02_solution_exploration.md)에서 도출된 아이디어 중,  
> MVP 단계에서 실제로 구현할 기능과 배제할 기능을 명확히 구분하기 위한 문서이다.  
> Cursor 및 개발자는 이 문서를 기준으로 기능 구현 우선순위를 판단한다.

---

## 1. 문서 목적

- MVP에서 실제로 구현할 기능을 명확히 정의한다.
- 각 기능이 어떤 문제를 해결하는지, 어떤 지표에 영향을 주는지 연결한다.
- 구현하지 않을 기능과 그 이유를 기록하여 범위 확장을 방지한다.

## 3. 결정로그 1: Store Loop 전환율 개선 (DRD-1)

### 3.1 결정 사항

Store Loop 전환율 개선을 위해 아래 3가지 기능을 통합 도입한다.

| 기능 | 목적 | 채택 여부 |
|------|------|-----------|
| 조건 기반 맞춤형 준비 항목 추천 | SP1 개선 | ✅ 채택 |
| 구조화된 탐색 결과 제공 | Store_R 감소, PB_t 감소 | ✅ 채택 |
| 즉시 저장 가능한 체크리스트 구조 | SP2 개선, Delay_t 감소 | ✅ 채택 |

---

### 3.2 선택 이유

`Search → Detail Check → Save` 구간의 전환 마찰을 동시에 낮추기 위함이다.

- **조건 기반 탐색** → Search 이후 첫 Detail Check까지의 탐색 시작 마찰을 줄여 SP1을 증가시킬 수 있음 (SP1 퍼널 관찰)
- **구조화된 결과 제공** → Detail Check에서의 반복 탐색을 줄여 여행 준비 부담도(PB_t)를 감소시킬 수 있음 (반복 탐색 관찰)
- **즉시 저장 구조** → Detail Check → Save까지의 행동 비용을 줄여 SP2 개선에 직접 기여함 (저장 전이 퍼널 관찰)

> 결과적으로 Store Loop의 `Store_R`, `Delay_t`, `PB_t`를 낮추고 `Store_LCP`를 높이는 방향으로 작동할 것으로 판단한다.

---

### 3.3 구현 범위 (MVP)

- 여행 조건 입력 UI (여행지, 일정 등 필터)
- 필터 기반 항목 조회 API 연동
- 구조화된 체크리스트 형태의 결과 제공
- 결과 항목에서 바로 Save 가능한 버튼
- Save 클릭 시 즉시 DB 저장

---

### 3.4 배제 사항

- 고도화된 추천 알고리즘 (ML/LLM 기반 개인화): 초기 데이터 부족, MVP 목표는 정확도보다 전환 구조 관찰
- 자동 분류 및 자동 완성: 동일 이유로 배제

---

### 3.5 관련 지표

| 기능 | 지표 | 공식 |
|------|------|------|
| 맞춤형 항목 추천 | SP1 | `P(DetailCheck \| Search)` |
| 구조화된 결과 제공 | Store_R | `P(ReSearch \| Search)` |
| 구조화된 결과 제공 | PB_t | 준비 부담 지수 |
| 즉시 저장 구조 | SP2 | `P(Save \| DetailCheck)` |
| 즉시 저장 구조 | Delay_t | 검색→저장 평균 소요 시간 |
| 공동 | Store_LCP | `SP1 × SP2` |
| 공동 | T_first(DetailCheck \| Search) | Search 후 첫 Detail Check까지 걸리는 시간 |
| 공동 | T_first(Save \| DetailCheck) | Detail Check 후 첫 Save까지 걸리는 시간 |

---

### 3.6 관찰 포인트 및 가설 폐기 기준

**⚠️ 하단 조건 만족 시 가설 폐기**

**Leading (선행 지표 - 조기 경보)**

| 조건 | 폐기 기준 |
|------|-----------|

| Detail Check 후 첫 Save 발생률 | 50% 미만 |

| Detail Check → 첫 Save까지 소요 시간 | 60초 초과 |

**Structural (구조 지표 - 목표 기준)**

- `Delay_t` 개선 여부

### 3.7 리스크 및 확장 계획

| 구분 | 내용 |

| ⚠️ 리스크 | 저장 구조가 복잡하면 오히려 첫 Save 전 이탈이 증가할 수 있음 |
| 🔮 확장 | 초기에는 핵심 조건 기반 탐색 + 단순 저장 구조를 우선 적용 → 이후 추천 정확도 및 자동 분류 기능으로 확장 |

## 4. 결정로그 2: Confirm Loop 전환율 개선 (DRD-2)

### 4.1 결정 사항

Confirm Loop 전환율 개선을 위해 아래 기능을 도입한다.

| 기능 | 목적 | 채택 여부 |
|------|------|-----------|
| 카테고리 기반 확인 구조 | CP1 ↑, CP2 ↑ | ✅ 채택 |
| 실행 순서 중심 체크리스트 | CP1 개선 | ❌ 배제 (2차 사이클 실험) |
| 누락 항목 추천 기능 | Backflow 감소 | ❌ 배제 (데이터 부족) |
| 휴지통 기능 | Backflow 감소 | ❌ 배제 (Backflow 포착 시 2차 사이클) |
| 준비 완료 체크 행동 강화 | CP2 ↑ | ❌ 배제 (행동 구조 파악 우선) |

---

### 4.2 선택 이유

`Saved List Open → Edit → Prepare Action` 전환을 강화하고 Backflow를 관찰하기 위함이다.

- **카테고리 기반 확인 구조**
  - `Saved List Open → Edit → Prepare Action`까지의 각 퍼널을 관찰 가능하게 함
  - 비구조적인 확인 행동을 줄이고 점검·수정·준비 행동으로의 전이를 증가시킬 수 있음

**배제 이유 상세:**

| 기능 | 배제 이유 |
|------|-----------|
| 실행 순서 중심 체크리스트 | 카테고리 구조 대비 우선순위 낮음. 추후 실험으로 반영 가능 |
| 누락 항목 추천 | 초기 데이터 불충분으로 추천 정확도 낮음, 효과 검증 어려움 |
| 휴지통 기능 | Backflow 발생 여부 먼저 확인 후 2차 사이클에서 도입 결정 |
| 준비 완료 체크 강화 | 사용자 행동 강화에 해당, 초기 행동 구조 파악 목적에 맞지 않음 |

---

### 4.3 구현 범위 (MVP)

- 체크리스트 카테고리 분류 (준비물 / 예약 / 기타 등)
- 카테고리별 리스트 UI 구성
- Edit 기능: 추가(Edit_add) / 삭제(Edit_del) / 텍스트 수정(Edit_text) / 순서 변경(Edit_reorder)

---

### 4.4 관련 지표

| 기능 | 지표 | 공식 |
|------|------|------|
| 카테고리 기반 확인 구조 | CP1 | `P(Edit \| SavedListOpen)` |
| 카테고리 기반 확인 구조 | CP2 | `P(PrepareAction \| Edit)` |
| 공동 | P(PrepareAction \| SavedListOpen) | SavedListOpen 대비 PrepareAction 도달률 |

---

### 4.5 관찰 포인트 및 가설 폐기 기준

|------|-----------|

| 카테고리별 점검 진입률 | ≤ 50% |

**Structural (구조 지표)**

| 지표 | 목표 기준 |
|------|-----------|

## 5. 결정로그 3: Total Loop 연결성 개선 (DRD-3)

### 5.1 결정 사항

Total Loop 연결성 개선을 위해 아래 기능을 도입한다.

| 기능 | 목적 | 채택 여부 |
|------|------|-----------|
| 준비 시작 진입 구조 | Travel Fixed → Store Loop 진입 개선 | ✅ 채택 |
| 저장-확인 루프 동시 수행 | Store → Confirm 연결 | ❌ 배제 (1차 사이클 후 A/B 테스트) |
| 진행도 시각화 | 반복 확인·Backflow 감소 | ❌ 배제 (후속 고도화) |
| Reuse 동기 설계 | 장기 재활성화 | ❌ 배제 (후속 고도화) |

---

### 5.2 선택 이유

`Travel Fixed → Store Loop → Confirm Loop` 구간의 연결 단절을 우선적으로 줄이기 위함이다.

- **준비 시작 진입 구조**
  - 여행 확정 이후 사용자가 다시 탐색부터 시작해야 하는 마찰을 줄임
  - 초기 진입 행동을 바로 Store Loop로 유도하여 `Store_LCP` 개선에 기여

**배제 이유 상세:**

| 기능 | 배제 이유 |
|------|-----------|
| 저장-확인 루프 동시 수행 | 초기 데이터 부족 상황에서 동시 제공 시 데이터 노이즈 발생 우려. 1차 사이클 후 A/B 테스트로 진행 예정 |
| 진행도 시각화 | 전체 루프의 첫 연결과 중간 연결 안정화가 우선. 시각화는 후속 고도화 단계에서 검토 |
| Reuse 동기 설계 | 초기 MVP에서 Reuse 포함 시 핵심 목표인 루프 연결성 관찰·완료 구조 검증이 흐려질 수 있음 |

---

### 5.3 구현 범위 (MVP)

- "여행 준비 시작하기" 진입 버튼
- 초기 체크리스트 생성 기능 (여행 추가 트리거)
- Travel Fixed 상태에서 바로 Store Loop로 연결되는 흐름

---

### 5.4 관련 지표

| 기능 | 지표 | 설명 |
|------|------|------|
| 준비 시작 진입 구조 | P(StoreLoop \| TravelFixed) | Travel Fixed 이후 Store Loop 진입률 |
| 준비 시작 진입 구조 | Travel Fixed 이후 첫 Search 발생률 | 진입 후 탐색 시작 여부 |
| 준비 시작 진입 구조 | Travel Fixed 이후 첫 Save 발생률 | 진입 후 저장 완료 여부 |

---

### 5.5 관찰 포인트 및 가설 폐기 기준

**⚠️ 하단 조건 만족 시 가설 폐기**

**Leading (선행 지표)**

| 조건 | 폐기 기준 |
|------|-----------|
| Travel Fixed 이후 첫 Search 발생률 | < 50% |
| Travel Fixed 이후 첫 Save 발생률 | < 30% |
| Save 이후 첫 Saved List Open 발생률 | < 50% |

**Structural (구조 지표)**

| 지표 | 목표 기준 |
|------|-----------|
| P(StoreLoop \| TravelFixed) | ≥ 50% |

**Lagging (후행 지표)**

- `Total_LCP` 상승 여부

> **주의:** 초기에는 지속성을 측정하기 어려우므로 코호트 분석보다 초기 진입 여부 관찰에 집중한다.  
> 코호트 등 장기 지표는 1차 사이클 이후 재검토한다.

---

### 5.6 리스크 및 확장 계획

| 구분 | 내용 |
|------|------|
| ⚠️ 리스크 | MVP에서 여러 연결 구조를 한 번에 통합하면 사용 흐름이 복잡해질 수 있음 |
| ⚠️ 리스크 | 저장 루프와 확인 루프를 한 구조에서 다룰 경우 이벤트 정의가 모호해지면 지표 비교 가능성 저하 |
| 🔮 확장 | 초기에는 Travel Fixed → Confirm Loop까지의 연결 안정화에 집중 → 이후 진행도 시각화, Reuse 동기 설계를 후속 사이클에서 검토 |
| 🧪 A/B 테스트 | 저장-확인 루프 동시 수행 편의성 기능은 1차 사이클 데이터 확보 후 2차 사이클에서 실험군으로 진행 |

## 6. 개발 참고 메모

### 기능 구현 시 판단 기준

```
기능을 추가하려 할 때 아래 질문을 먼저 확인한다.

1. 이 기능이 채택된 결정로그 항목에 포함되어 있는가?
2. 포함되지 않는다면, 배제 이유를 확인하고 재검토 시점이 맞는지 판단한다.
3. MVP 범위를 벗어나는 기능은 바로 구현하지 않고 별도 후보로만 기록한다.
```

### 사이클별 재검토 로드맵

| 사이클 | 재검토 대상 |
|--------|-------------|
| 1차 MVP | 준비 시작 진입 + 조건 기반 탐색 + 즉시 저장 + 카테고리 확인 구조 |
| 2차 사이클 | 휴지통 기능 (Backflow 포착 시), 저장-확인 루프 A/B 테스트 |
| 3차 사이클 이후 | 진행도 시각화, 누락 항목 추천, Reuse 동기 설계, 준비 완료 체크 강화 |

---

## 7. 이 문서의 핵심

> MVP는 모든 문제를 해결하는 단계가 아니다.  
>
> **지금 단계의 목표는 딱 하나다:**  
> `Travel Fixed → Store Loop → Confirm Loop` 흐름이 실제로 연결되는지 관찰하고,  
> 그 과정에서 발생하는 병목을 측정 가능한 상태로 만드는 것이다.
>
> 배제된 기능은 "나쁜 아이디어"가 아니라 "지금 단계가 아닌 것"이다.
