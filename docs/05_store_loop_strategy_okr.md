# Store Loop Strategy OKR

> **문서 용도:** 이 문서는 Store Loop 전략 목표와 기대 결과를 개발자 관점에서 정리한 참고 문서다.  
> 기능을 구현할 때 "이 기능이 왜 필요한지", "어떤 지표에 영향을 주는지"를 빠르게 확인하기 위해 작성됐다.  
> 관련 결정 근거는 `03_decision_log.md`, 이벤트 수집 구조는 `04_meta_okr.md`를 함께 참고한다.

---

## 1. 문서 목적

### 이 문서가 필요한 이유

기능을 개발할 때 "어떤 기능을 만드는지"만 알면 방향을 잃기 쉽다.  
이 문서는 **Store Loop에서 무엇을 달성하려 하는지**, **왜 그 기능이 필요한지**를 연결해준다.

### 어떤 의사결정에 참고되는가

- **기능 추가 판단:** 새 기능이 Store Loop의 어느 지표를 개선하는지 확인할 때
- **컴포넌트 설계:** 탐색 → 상세 확인 → 저장 흐름을 끊기지 않게 설계할 때
- **이벤트 설계:** 어떤 사용자 행동을 수집해야 하는지 결정할 때
- **우선순위 판단:** MVP에서 먼저 만들 기능과 나중으로 미룰 기능을 구분할 때

## 2. Store Loop 정의

### Store Loop가 무엇인가

Store Loop는 사용자가 **여행 준비 정보를 탐색하고 체크리스트에 저장하는 과정 전체**를 말한다.

```
Search → Detail Check → Save
```

- **Search:** 여행 조건(여행지, 일정 등)을 기반으로 필요한 준비 항목을 탐색하는 단계
- **Detail Check:** 탐색된 항목의 내용을 확인하고 필요 여부를 판단하는 단계
- **Save:** 확인한 항목을 내 체크리스트에 저장하는 단계

### 전체 서비스 흐름에서 어디에 해당하는가

```
Travel Fixed
 → [Store Loop: Search → Detail Check → Save]   ← 이 문서의 범위
 → [Confirm Loop: Saved List Open → Edit → Prepare Action]
 → Reuse
```

Store Loop는 여행이 확정된 직후 시작되는 **첫 번째 준비 행동 구간**이다.  
Store Loop가 안정적으로 작동하지 않으면 Confirm Loop 자체가 성립하지 않는다.

---

## 3. 해결하려는 문제

사용자는 여행 준비 정보를 탐색하더라도 바로 저장으로 이어지지 않는다.  
아래는 Store Loop에서 실제로 발생하는 구체적인 문제들이다.

| 문제 | 설명 |
|------|------|
| **반복 탐색** | 여러 채널(블로그, SNS, 공식사이트)에서 같은 정보를 반복해서 찾는다 |
| **저장 지연** | 탐색한 정보를 정리하는 과정이 번거로워 저장까지 시간이 오래 걸린다 |
| **정보 정리 부담** | 탐색 결과가 구조화되어 있지 않아 사용자가 직접 분류하고 정리해야 한다 |
| **상황 미반영** | 여행지, 동행인, 일정에 따라 필요한 항목이 다르지만 일반화된 결과만 제공된다 |

이 문제들이 누적되면 사용자는 **Store Loop를 완료하지 못하고 이탈하거나 반복 탐색을 계속**하게 된다.

---

## 4. 전략 목표

Store Loop 전략의 핵심은 단 하나다.

> **사용자가 탐색을 시작하면, 반복 없이 저장까지 이어지게 만드는 것**

구체적으로는 아래 행동 변화를 만들고자 한다.

- 탐색을 시작한 사용자가 항목 상세를 **빠르게 확인**할 수 있어야 한다 → SP1 개선
- 상세를 확인한 사용자가 **즉시 저장**할 수 있어야 한다 → SP2 개선
- 이미 탐색한 내용을 **다시 찾지 않아도 되어야** 한다 → Store_R 감소
- 탐색 시작부터 저장까지 **오래 걸리지 않아야** 한다 → Delay_t 감소

## 5. Strategy OKR

### Objective

> **사용자가 자신의 여행 조건에 맞는 준비 정보를 빠르게 탐색하고,  
> 구조화된 확인을 거쳐 저장까지 이어지는 행동 구조를 형성한다.**

쉽게 말하면: _"탐색하면 저장까지 자연스럽게 이어지는 흐름을 만든다"_

---

### Key Results

MVP 단계에서 확인하고자 하는 4가지 핵심 결과다.

#### K1. `Search → Detail Check` 흐름이 실제로 발생하는지 확인한다

- 사용자가 탐색을 시작한 뒤 실제로 항목을 클릭해서 확인하는지 보는 것
- 이게 안 일어난다면: 탐색 결과 자체가 사용자의 흥미를 끌지 못하고 있다는 신호
- **수집 이벤트:** `search_start` → `detail_check_open`

#### K2. 구조화된 탐색 결과가 반복 탐색을 줄이는지 확인한다

- 한 번 탐색한 사용자가 다시 같은 내용을 검색하는지 보는 것
- 재탐색이 계속 발생한다면: 결과가 충분히 구조화되지 않았거나 정보가 부족하다는 신호
- **수집 이벤트:** `research_trigger`

#### K3. `Detail Check → Save` 전환 흐름이 실제로 발생하는지 확인한다

- 항목을 확인한 사용자가 실제로 저장 버튼을 누르는지 보는 것
- 이게 안 일어난다면: 저장 행동까지의 마찰이 너무 크거나 저장 가치가 느껴지지 않는다는 신호
- **수집 이벤트:** `detail_check_open` → `save_complete`

#### K4. 저장 루프의 시간 지연과 행동 부담이 어느 구간에서 발생하는지 식별한다

- 어느 단계에서 오래 걸리는지, 어느 단계에서 이탈하는지 보는 것
- 병목 구간을 찾아야 다음 개선 방향을 잡을 수 있다
- **수집 이벤트:** 각 이벤트의 타임스탬프 차이

---

### Result / 기대 상태

이 전략이 잘 작동했을 때 사용자 행동은 이렇게 바뀐다.

| 결과 지표 | 이 전략이 잘 작동한 상태 |
|-----------|--------------------------|
| `SP1` | 탐색 후 항목 상세 확인이 자주 발생함 |
| `Store_R` | 같은 내용을 재탐색하는 빈도가 줄어듦 |
| `SP2` | 상세 확인 후 저장까지 이어지는 비율이 높음 |
| `Delay_t` | 탐색 시작부터 저장까지 소요 시간이 짧음 |
| `Store_LCP` | 전체 저장 루프(탐색→확인→저장)가 안정적으로 닫힘 |
| `PB_t` | 준비 부담 지수가 낮아짐 (재탐색·지연·정보 공백 감소) |

## 6. 핵심 기능 전략

MVP에서 채택된 3가지 기능이 Store Loop 어디에 영향을 주는지 정리한다.

### 6.1 조건 기반 탐색 (맞춤형 항목 추천)

**영향 지점:** `Search → Detail Check` 구간 (SP1 개선)

- **왜 필요한가:** 일반적인 검색 결과는 사용자의 여행 조건(여행지, 동행인 등)과 맞지 않아 반복 탐색이 발생한다.
- **무엇을 하는가:** 여행 조건을 입력하면 그에 맞는 준비 항목을 필터링해서 보여준다.
- **기대 효과:** 탐색 시작 후 상세 확인 전환율(SP1) 증가, 재탐색 비율(Store_R) 감소

```
구현 포인트 (TripNewPage.jsx → TripSearchPage.jsx)
- 여행지, 출발일, 귀국일, 동행 유형 필터 입력
- 필터 값을 API 파라미터로 전달
- 조건에 맞는 구조화된 결과 반환
```

### 6.2 구조화된 탐색 결과 제공

**영향 지점:** `Detail Check` 구간 (반복 탐색 감소)

- **왜 필요한가:** 텍스트 덩어리로 된 정보는 사용자가 직접 분류하고 판단해야 해서 피로도가 높다.
- **무엇을 하는가:** 탐색 결과를 카테고리, 설명, 저장 버튼이 있는 구조화된 카드 형태로 제공한다.
- **기대 효과:** 정보 정리 부담 감소, Delay_t 감소, Store_R 감소

```
구현 포인트 (TripSearchPage.jsx - SearchResultItem)
- 카테고리 태그 표시 (서류/준비물/예약/건강 등)
- 항목명 + 간단한 설명 구조
- 저장 상태(저장됨 / 저장) 즉시 피드백
```

### 6.3 즉시 저장 가능한 체크리스트 구조

**영향 지점:** `Detail Check → Save` 구간 (SP2 개선)

- **왜 필요한가:** 탐색 결과를 확인했어도 저장하려면 별도 과정을 거쳐야 한다면 행동 비용이 높아진다.
- **무엇을 하는가:** 항목 카드에서 클릭 한 번으로 즉시 내 체크리스트에 저장된다.
- **기대 효과:** SP2 증가, Delay_t 감소

```
구현 포인트 (TripSearchPage.jsx - handleSave)
- 저장 버튼 클릭 → 즉시 API 호출 → UI 상태 즉시 반영
- 저장 완료 시 "저장됨 ✓" 표시 (사용자 피드백 즉시 제공)
- TODO: save_complete 이벤트 수집
```

## 7. 핵심 지표

지표가 처음이라면 "어떤 행동을 보는 지표인가"를 먼저 이해하는 것이 중요하다.

| 지표 | 쉬운 설명 | 공식 | 이 지표가 낮으면 |
|------|-----------|------|-----------------|
| `SP1` | 탐색한 사람 중 실제로 항목을 확인한 비율 | `Detail Check / Search` | 탐색 결과가 사용자의 흥미를 끌지 못함 |
| `SP2` | 항목을 확인한 사람 중 실제로 저장한 비율 | `Save / Detail Check` | 저장까지의 마찰이 크거나 저장 가치가 낮음 |
| `Store_LCP` | 탐색 시작 → 저장까지 전체 흐름이 완성된 비율 | `SP1 × SP2` | Store Loop 전체가 닫히지 않음 |
| `Store_R` | 같은 준비 과정에서 재탐색이 발생한 비율 | `ReSearch / Search` | 한 번 탐색으로 필요한 정보를 다 얻지 못함 |
| `Delay_t` | 탐색 시작 → 저장까지 걸린 평균 시간 | 타임스탬프 차이 (정규화) | 사용자가 결정하는 데 너무 오래 걸림 |
| `PB_t` | 전체 준비 부담도 (재탐색 + 정보 공백 + 지연의 합산) | `α·Store_R + β·IG_t + γ·Delay_t` | 준비 과정 전체가 사용자에게 부담으로 느껴짐 |

### 지표 해석 흐름

```
SP1 낮음  → 탐색 결과 품질 문제  → 조건 기반 탐색 개선
SP2 낮음  → 저장 마찰 문제      → 즉시 저장 구조 개선
Store_R 높음 → 반복 탐색 문제   → 구조화된 결과 개선
Delay_t 높음 → 결정 지연 문제   → 탐색-저장 UX 단순화
```

## 8. MVP 구현 우선순위

### 지금 바로 구현할 것 (1차 MVP)

| 기능 | 구현 파일 | 이유 |
|------|-----------|------|
| 여행 조건 입력 폼 | `TripNewPage.jsx` | Travel Fixed → Store Loop 진입 마찰 제거 |
| 카테고리 필터 탭 | `TripSearchPage.jsx` | SP1 측정을 위한 최소 탐색 구조 |
| 구조화된 결과 카드 | `TripSearchPage.jsx` | Detail Check 이벤트 수집 가능하게 |
| 즉시 저장 버튼 | `TripSearchPage.jsx` | SP2 측정 및 Delay_t 감소 |
| 이벤트 수집 연동 | `TripSearchPage.jsx` | 지표 계산 가능하게 만들기 |

### 나중으로 미룰 것 (2차 사이클 이후)

| 기능 | 배제 이유 |
|------|-----------|
| ML/LLM 기반 추천 | 초기 데이터 부족, 정확도 보장 어려움 |
| 자동 분류 / 자동 완성 | 동일 이유, 구조 관찰이 우선 |
| 검색어 직접 입력 탐색 | 조건 기반 탐색 효과 측정이 먼저 |

### 왜 이렇게 나누는가

MVP 단계의 목표는 **"기능을 많이 만드는 것"이 아니라 "루프가 실제로 작동하는지 관찰하는 것"** 이다.  
지금은 `Search → Detail Check → Save` 이벤트가 실제로 수집되는지 확인하는 것이 먼저다.  
이벤트가 충분히 쌓이면 그때 데이터 기반으로 다음 기능을 결정한다.

## 9. 개발 시 유의사항

### 이벤트 수집

Store Loop의 모든 전환 지점에는 이벤트가 수집되어야 한다.  
이벤트가 없으면 지표를 계산할 수 없고, 지표가 없으면 개선 방향을 잡을 수 없다.

```javascript
// TripSearchPage.jsx 내 이벤트 수집 위치

// search_start: 페이지 진입 시 or 카테고리 변경 시
// detail_check_open: 항목 카드를 열거나 상세를 확인할 때
// save_complete: 저장 버튼 클릭 후 API 성공 응답 시
// research_trigger: 동일 세션 내 재탐색(필터 재변경) 시
```

> **주의:** 이벤트 이름은 한 번 정하면 변경하기 어렵다.  
> 변경하면 과거 데이터와 비교할 수 없게 된다. 반드시 `04_meta_okr.md`의 이벤트 정의를 먼저 확인할 것.

### 상태 관리

- 저장 상태(`savedIds`)는 낙관적 업데이트(Optimistic Update)로 처리한다.
  - 저장 버튼 클릭 즉시 UI에 "저장됨"을 표시하고, API 실패 시 롤백
  - 이렇게 해야 `Delay_t`가 짧게 측정된다 (사용자가 기다리지 않아야 함)
- 재탐색 감지: 동일 세션에서 필터를 다시 변경하는 행동 = `research_trigger`

### API 연결

```javascript
// 연결해야 할 API 엔드포인트 (추후 백엔드 연동)
// GET  /api/trips/:id/search?destination=...&companions=...   → 조건 기반 항목 조회
// POST /api/trips/:id/checklist                              → 항목 즉시 저장
// POST /api/events                                           → 이벤트 로그 전송
```

### Store Loop와 Confirm Loop는 반드시 분리된 페이지로 유지

- `/trips/:id/search` (Store Loop) ↔ `/trips/:id/checklist` (Confirm Loop)
- 두 루프를 같은 페이지에 탭으로 합치면 이벤트 집계가 섞인다.
- `re_store_trigger` 이벤트가 발생해야 Backflow를 측정할 수 있는데, 같은 페이지면 측정이 불가능하다.
