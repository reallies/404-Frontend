## 결정로그 1: 저장 루프(Store Loop) 전환율 개선

| 항목 | 내용 |
| --- | --- |
| 결정 사항 | **Store Loop 전환율 개선을 위해 조건 기반 탐색 + 탐색 결과 구조화 + 즉시 저장 가능한 체크리스트 구조를 통합 도입**

1. SP1 개선을 위해 [사용자 필터 맞춤형 준비 항목 추천] 도입 결정
2. Detail Check에서 발생하는 반복탐색 문제를 해결하기 위해 [구조화된 탐색 결과 제공]을 도입 결정
3. SP2 및 전이 지연 문제($Delay_t$) 개선을 위해 [즉시 저장 가능한 체크리스트 구조] 도입 결정 |
| 대안 | 1. 맞춤형 준비 항목 추천 (채택)
2. 탐색 결과 구조화하여 제공 (채택)
3. 즉시 저장 가능한 체크리스트 구조 도입 (채택) |
| 이유 | **Search → Detail Check → Save 구간의 전환 마찰을 동시에 낮추기 위함**

1. **필터 기반 맞춤형 준비 항목 추천은
  •** Search에서 Detail Check까지의 퍼널 관찰하기 위함
  • 탐색 시작 마찰을 줄여 SP1을 증가시킬 수 있음
2. **구조화된 탐색 결과 제공**
  • Detail Check에서의 반복 탐색을 관찰하기 위함
  • 반복 탐색을 감소하여 여행준비부담도를 감소시킬 수 있음

3. **즉시 저장 가능한 구조**
  • Detail Check 이후 Save까지의 퍼널 관찰하기 위함
  **•** 행동비용을 줄여 전이 지연을 개선하고 SP2 개선에 직접 관여함

⇒ 결과적으로 Store Loop에 대한 Store_R, Delay_t, PB_t를 낮추고 Store_LCP를 높이는 방향으로 작동할 것으로 판단함 |
| 관련 지표 | 1. **필터 기반 맞춤형 준비 항목 추천
  •** `P(DetailCheck | Search) = SP1`
****2. **구조화된 탐색 결과 제공**
  • `P(ReSearch | Search) = Store_R`
  **•** $PB_t$
****3. **즉시 저장 가능한 구조
  •** `P(Save | DetailCheck) = SP2` 
  • $Delay_t$  |
| 관찰 포인트 | **하단 조건 만족 시 가설 폐기

Leading:** 
Search 후 첫 Detail Check 발생률 50% 미만 시
Detail Check 후 첫 Save 발생률 50% 미만 시
Search 후 첫 Detail Check까지 걸리는 시간 60초 초과 시
Detail Check 후 첫 Save까지 걸리는 시간 60초 초과 시 

**Structural:** 
SP1 ≤ 70%
SP2 ≤ 50%
Search 후 Detail Check까지 평균 시간 ≤  180초
Detail Check 후 Save까지 평균 시간 ≤ 180초
검색 착수율(SS) ≥ 20%
저장 전환율(SCR) ≥ 70%

**Lagging:** 
Store_LCP 상승 여부
Store_R 감소 여부
Delay_t 개선 여부 |
| 메모 (리스크 및 확장) | • 리스크: 필터 조건이 너무 적거나 부정확하면 원하는 결과를 충분히 제공하지 못할 수 있음
• 리스크: 저장 구조가 복잡하면 오히려 첫 Save 전 이탈이 증가할 수 있음
• 확장 계획: 초기에는 핵심 조건 기반 탐색 + 단순 저장 구조를 우선 적용하고, 이후 추천 정확도 및 자동 분류 기능으로 확장 가능 |

## 결정로그 2: 확인 루프(Confirm Loop) 전환율 개선

| 항목 | 내용 |
| --- | --- |
| 결정 사항 | **Confirm Loop 전환율 개선을 위해 카테고리 기반 확인 구조 + ~~삭제/복구 구조를 통합~~ 도입**
• Saved List Open 이후 단순 열람이 아니라 Edit 행동으로 자연스럽게 이어지도록 확인 구조를 제공함
• Prepare Action까지 쉽게 이어지게 함

1. CP1, CP2 개선 및 준비 행동으로의 전이 마찰을 줄이기 위해 [카테고리 기반 확인 구조] 도입 결정 |
| 대안 | **1. 실행 순서 중심 체크리스트 구조 제공(배제)**
  • 카테고리 기반 구조보다 우선순위가 낮아 MVP 반영에서 배제, 추후 실험 목적으로 다른 사이클에 반영 가능
2. **카테고리 기반 확인 구조 도입(채택)**
3. **누락 항목 추천 기능(배제)**
  • 초기 단계에서 데이터가 충분치 않아 추천 알고리즘에 대한 구현 가능성이 낮음
  • 또한 효과 검증이 어렵고 효과 크기도 제한적임으로  MVP에서 우선 배제 처리
4. **휴지통 기능(배제)**
  • 초기 MVP에서 배제, Backflow 발생이 포착되면 제2 사이클에서 기능 추가
5. **준비 완료 체크 행동 강화(배제)**
  • 사용자 행동 강화에 해당하므로, 초기 행동 구조 파악 목적에 맞지 않아 초기 MVP 배제 |
| 이유 | **Saved List Open → Edit → Prepare Action 전환을 강화하고 Backflow를 줄이기 위함

1. 카테고리 기반 확인 구조 도입**
  • Saved List Open → Edit → Prepare Action까지의 각 퍼널 관찰 가능
  • 비구조적인 확인 행동을 줄이고 점검/수정 + 준비 행동으로의 전이를 증가시킬 수 있음 |
| 관련 지표 | 1. **카테고리 기반 확인 구조**
`P(Edit | SavedListOpen) = CP1`
`P(PrepareAction | Edit) = CP2`

**공동지표 :**
`P(PrepareAction | SavedListOpen)` Saved List Open 대비 Prepare 
Action 도달률 |
| 관찰 포인트 | **하단 조건 만족 시 가설 폐기

Leading:**
Saved List Open 후 첫 Edit 발생률 ≤ 30%
Edit 후 첫 Prepare Action 발생률 ≤ 20%
카테고리별 점검 진입률 ≤ 50%

**Structural:** 
CP1 ≤ 70%
CP2 ≤ 50%
준비 완료까지 걸리는 시간

**Lagging:**
Saved List Open 대비 Prepare Action 도달 여부
저장 이후 Saved List Open 전환 여부 |
| 메모 (리스크 및 확장) |   • 리스크: 카테고리 항목이 너무 많아지면 확인 루프 자체가 복잡해질 수 있음
  • 확장 계획: 초기에는 카테고리 기반 구조만 우선 적용하고, 이후 데이터 축적 후 누락 항목 추천 또는 준비 완료 체크 구조를 후속 사이클에서 검토 가능
  • 잔여 여행 기간별 Edit 사용률 추후 관찰 포인트로 추가 |
